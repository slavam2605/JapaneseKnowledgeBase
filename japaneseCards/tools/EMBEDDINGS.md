# Локальный индекс карточек + эмбеддинги

Оффлайн-индекс всех японских колод (архив `0. Архив::日本語::*` + новые `新日本語::*`) для
семантического поиска и поиска дублей, чтобы LLM не грузила тысячи карточек в контекст.

- **База:** `data/anki_index.sqlite3` (в `.gitignore`, пересобираемая). Таблицы: `anki_notes`
  (очищенные поля), `note_embeddings` (векторы BLOB), `export_runs` / `embedding_runs` (аудит).
- **Модель:** локальная `BAAI/bge-m3` (1024-dim, мультиязычная ja+ru), через `.venv` (Python 3.14,
  `sentence-transformers` + `torch`). Модель кэшируется в `~/.cache/huggingface` (общая с greekCards).
- **Код:** переиспользован из `greekCards/scripts` — `embedding_utils.py`, `build_embeddings.py`,
  `search_similar.py` почти без правок; `export_anki_notes.py` переписан под японские note types.
- **Масштаб:** ~4283 заметки. Поиск — брутфорс-косинус в numpy, векторной БД не нужно.
- **`embedding_text` — намеренно скудный:** `headword（reading） meaning`, без примеров/меток
  (`build_embedding_text` в `export_anki_notes.py`). A/B против варианта с примерами и метками
  `意味:/例:` показал: скудный лучше по всем метрикам — косинус истинного дубля 0.88→0.95, дублей
  на 1-м месте 81%→88%, отрыв top1−top2 0.08→0.15 (надёжнее порог), омонимы разъезжаются
  (零(れい)↔例(れい) 0.60→0.40). Примеры зашумляют сигнал. Богатый текст (чтения, примеры) остаётся
  в `back_clean` для показа, но не индексируется. Пробел перед meaning — только для читаемости.

## Разовая / обновляющая сборка

```bash
cd japaneseCards
export HF_HUB_OFFLINE=1
# 1) выгрузить колоду в SQLite (read-only AnkiConnect). Kanji writing НЕ грузить — те же заметки, что Kanji.
.venv/bin/python tools/export_anki_notes.py --deck '0. Архив::日本語::Words::Words MNN1-2'
# 2) построить эмбеддинги (backend local = bge-m3 по умолчанию; только новые/изменённые заметки)
.venv/bin/python tools/build_embeddings.py --deck '0. Архив::日本語::Words::Words MNN1-2'
```

Экспорт идемпотентен (upsert по content_hash, пропавшие заметки помечаются `is_present=0`);
build — инкрементальный (`--force` для пересчёта). Список всех колод — в `README.md`/`STATUS.md`.

**Два хеша (важно):** `content_hash` = хеш всех полей заметки **включая теги** (для учёта
inserted/updated в экспорте); `embedding_hash` = хеш **только `embedding_text`**. Пересборка
эмбеддингов завязана на `embedding_hash`, поэтому смена тегов (`to_remove`,
`ai_generated::to_review`…) НЕ вызывает пересчёт векторов — только реальное изменение
embedding_text. `note_embeddings.source_content_hash` хранит именно `embedding_hash`.

## Поиск (use case 1: LLM ищет в колоде)

```bash
export HF_HUB_OFFLINE=1
# по свободному тексту (русский/японский), по всему архиву:
.venv/bin/python tools/search_similar.py --text "идёт дождь, погода, зонт" \
    --deck-prefix '0. Архив::日本語' --top 5
# по конкретной заметке:
.venv/bin/python tools/search_similar.py --note-id <id> --query-deck '新日本語::Слова' \
    --deck-prefix '0. Архив::日本語' --top 5
```

`--deck` — одна точная колода; `--deck-prefix` — все колоды с этим префиксом (весь архив сразу).

## Поиск дублей для удаления (use case 2)

Для новой карточки найти покрывающие её старые в архиве:
`--note-id <новая> --query-deck '新日本語::Слова' --deck-prefix '0. Архив::日本語'`.

Порог: косинус **≥ ~0.90** = почти наверняка дубль (напр. 駐車場↔駐車場 = 0.93).

**Два слоя, дополняющие друг друга (см. решение в памяти archive-index-plan):**
1. Детерминированный: точное совпадение `front_normalized` (нормализованное чтение каны).
   Дёшево сужает кандидатов. **Caveat:** чтение НЕ уникально — `零(れい)` ≠ `例(れい)`. Поэтому
   обязательно подтверждать совпадением кандзи (`front_terms_json`) ИЛИ семантикой.
2. Семантический: косинус эмбеддингов. Отсекает омонимы-по-чтению (零 vs 例 далеки), ловит
   орфографические варианты (面白い/おもしろい).

> **Перед дедупом переэкспортировать архив.** Индекс — снимок; удаление карты в Anki (пользователь
> чистит `to_remove` между проходами) **не** трогает SQLite, строка висит с `is_present=1`, и скан
> матчит призраков. Переэкспорт архивных колод (`export_anki_notes.py --deck …` по каждой) выставит
> `is_present=0` на удалённых — скан их фильтрует. Дёшево (read-only, секунды); `build_embeddings`
> для чистых удалений — no-op. Скилл `dedup-cards` делает это в шаге 1.

**Батч-скан для всего набора новых карт — `dedup_scan.py`** (read-only, ничего не пишет в Anki):
```bash
.venv/bin/python tools/dedup_scan.py            # только новые без тега ai_generated::deduped
.venv/bin/python tools/dedup_scan.py --all      # пересканировать всё
```
Для каждой новой заметки берёт top-N по косинусу **∪** все заметки с совпадающим нормализованным
чтением (дубль не теряется, даже если косинус просел — `おじいさん` cos 0.78). Классифицирует по
**поверхностной форме**: `SURE` (чтение+запись совпали → дубль, косинус вторичен), `REVIEW`
(орфо-вариант), `HOMOPHONE` (совпало только чтение, cos<0.80 → скорее разные слова `例`/`零`),
`SEMANTIC` (разное чтение, высокий косинус → не дубль). Пишет `data/removal_candidates.json` +
`data/removal_candidates_review.md`. Применяет решения (перенос картинки, `to_remove`,
`ai_generated::deduped`) уже скилл `dedup-cards`.

Агент **никогда не удаляет** старые карточки, даже с явным подтверждением пользователя (см. правила
безопасности в `knowledge/conventions.md`). Подтверждённого кандидата можно только пометить
`to_remove` и заморозить; физически удалить его позже может пользователь вручную. Инструмент только
предлагает кандидатов.
